\section{Data Storage Logic}
Efficiently storing and retrieving agents is critical for performance. The framework uses a **Structure-of-Arrays (SoA)** approach mixed with object-oriented containers to balance cache efficiency and code maintainability.

\subsection{Design Choice: Hybrid Storage}
Imagine a hospital database. If every patient's record is stored as an individual physical folder scattered across different filing cabinets, finding all patients of a certain age requires opening and reading through every single folder, which takes a lot of time. In computer science, this is like storing each agent as an independent object in memory (which causes slow memory access, or "cache misses").

Alternatively, we could write all ages in one long list, all locations in another, and all names in a third. This is incredibly fast for the computer to search, but it makes the code hard for human researchers to write and read, as you constantly have to coordinate between different lists.

To solve this, our simulation uses a **hybrid design**:
\begin{itemize}
    \item \textbf{For the computer}: All data is stored in unified, contiguous memory blocks (a "Structure of Arrays" or SoA) inside a central \texttt{world\_container} for maximum speed.
    \item \textbf{For the researcher}: The code exposes individual agents as if they were simple, familiar folders (\texttt{Agent} types) containing their own attributes. A pointer mechanism links these folders directly to the fast database tables under the hood, giving us the best of both worlds: human-readable code and computer-optimal speed.
\end{itemize}

\subsection{The Agent Array}
Agents are stored in a large, pre-allocated 2D array within the \texttt{world\_container}:
\begin{lstlisting}[language=Fortran]
type(Agent), allocatable :: agents(:,:) ! (max_size, npops)
\end{lstlisting}
\begin{itemize}
    \item \textbf{Rows}: Iterate through individual agents (up to \texttt{max\_size}).
    \item \textbf{Columns}: Separate different populations (e.g., \texttt{npops}).
\end{itemize}
This allows for contiguous memory access when iterating over a single population, which is the most common operation.

\subsection{Spatial Indexing: The Grid}
For spatial interactions (such as mating, resource competition, or cultural exchange), agents need to know who is nearby. Checking every single agent in the entire world to see if they are close is extremely slow ($O(N^2)$ complexity). 

Instead, the framework indexes agents by their geospatial location using a 2D \texttt{Grid} structure:
\begin{itemize}
    \item \textbf{Analogy}: Think of a massive chessboard where every square (grid cell) keeps a small list of pieces currently standing on it.
    \item \textbf{Mechanics}: Each cell (\texttt{grid\_cell}) contains a dynamic list of Agent IDs. When an agent moves from one coordinate to another, it is simply crossed off the old cell's list and added to the new one.
\end{itemize}
This allows any module to immediately find all neighbors by checking only the pieces standing on the same or adjacent board squares.


\subsubsection{Module Types: Agent-Centric vs Cell-Centric}
When developing modules, you must decide the primary access pattern:
\begin{itemize}
    \item \textbf{Agent-Centric Modules}: Functions that apply to a specific agent (e.g., Movement). These iterate over the \texttt{agents} array.
    \begin{lstlisting}[language=Fortran]
    subroutine agent_move(agent_ptr)
        type(Agent), pointer :: agent_ptr
    \end{lstlisting}
    \item \textbf{Cell-Centric Modules}: Functions that apply to a location (e.g., Overpopulation Death). These iterate over the \texttt{grid}.
    \begin{lstlisting}[language=Fortran]
    subroutine death_overpopulation(cell_ptr)
        type(cell), pointer :: cell_ptr
    \end{lstlisting}
\end{itemize}

\section{Global State \& Metrics}
In addition to individual agent data, the \texttt{world\_container} tracks macroscopic simulation states and per-tick accumulators to monitor population health and system performance.

\subsection{Tick Accumulators (\texttt{t\_tick\_accumulators})}
A history of accumulators is maintained (\texttt{accumulators\_history(:)}) to track events within a single tick. These are automatically reset at the start of each step:
\begin{itemize}
    \item \texttt{phi\_birth\_acc / phi\_death\_acc}: Cumulative birth and death counts.
    \item \texttt{n\_alive\_acc}: Count of agents currently alive.
    \item \texttt{ms\_per\_tick}: Execution time for the current simulation step.
\end{itemize}

\subsection{Dynamic State (\texttt{t\_dynamic\_state})}
Global variables that evolve over time and influence agent behavior:
\begin{itemize}
    \item \texttt{K\_fertility}: A macroscopic scaling factor used to regulate population growth according to the Verhulst model.
\end{itemize}

\section{Module \texttt{mod\_agent\_world}}
This module defines the central \texttt{world\_container} which holds:
\begin{itemize}
    \item \texttt{agents}: The main agent array.
    \item \texttt{grid}: The spatial grid.
    \item \texttt{index\_map}: The ID lookup hashmap.
    \item \texttt{config}: Global configuration parameters.
    \item \texttt{accumulators\_history}: Buffer of recent performance and biological metrics.
    \item \texttt{dynamic\_state\_vars}: Current macroscopic simulation state.
\end{itemize}
It serves as the "Hub" for all data access.

\subsection{Agent Lookup: Hashmap}
Sometimes, a simulation module needs to find a specific agent immediately using its unique identification number (for example, when a child needs to locate its mother). Searching through millions of memory records one-by-one until you find the right ID would stall the simulation.

To prevent this, the system uses a custom integer hashmap (\texttt{mod\_hashmap}):
\begin{itemize}
    \item \textbf{Analogy}: Think of this as the index at the back of a thick textbook. Instead of reading the whole book to find a mention of a term, you look it up in the index to find the exact page number.
    \item \textbf{Mechanics}: The hashmap takes the Agent ID as a "Key" and instantly returns a "Value" containing the exact coordinates (the population index and list index) of where that agent resides in the database.
\end{itemize}
This allows instant ($O(1)$) access to any agent's data.

\section{Compaction}
When agents die, they are marked with \texttt{is\_dead = .true.}. If we immediately deleted them and shifted the remaining millions of agents down in memory to fill the gap, it would require a massive amount of moving work, slowing the simulation down to a crawl.

Instead, the system leaves them in place temporarily and cleans them up all at once at the end of each tick using a process called **Compaction**:
\begin{itemize}
    \item \textbf{Analogy}: Imagine a classroom where some students have left. Instead of shifting everyone over one-by-one, we ask the students sitting at the very back of the room to move up and fill the empty desks at the front.
    \item \textbf{Mechanics}: The \texttt{compact\_agents} subroutine reads the agent array from both ends simultaneously. When it finds a "dead" desk on the left, it moves a "living" agent from the rightmost active index into it.
    \item \textbf{Result}: The array is cleanly packed at the front, and the active population size is logically decreased without the computer needing to reallocate or shift large amounts of data.
\end{itemize}


\section{Density Analysis \& Clustering}
The simulation includes a suite of spatial analysis modules (\texttt{mod\_watershed}, \texttt{mod\_kmeans}, \texttt{mod\_clustering}) that group grid cells into clusters based on agent density and HEP suitability.

\subsection{Global Density Smoothing}
To ensure consistency across different clustering algorithms, a unified density surface (\texttt{human\_density\_smoothed}) is generated once per tick. This surface is processed using an iterative box filter.

\subsection{Algorithms}
\begin{enumerate}
    \item \textbf{Watershed}: Performs gradient ascent on the global density surface.
    \item \textbf{K-Means++}: Utilizes a \textbf{Farthest-First Traversal} initialization to ensure centroids are placed in remote agent populations, avoiding redundant clustering on a single peak.
    \item \textbf{Convex Hull Fill}: A post-processing step that fills internal holes in clusters, ensuring structural integrity for visualization and demographic analysis.
\end{enumerate}
